# v0.1.0-alpha.1 Release Notes

## Overview

This is the first public alpha of Polykube — a Kubernetes operator for multicluster workload placement and runtime wiring. It targets engineering teams running two or more Kubernetes clusters who want declarative, GitOps-compatible control over where workloads run and how they route traffic across cluster boundaries.

## What works in this release

### Core operator controllers

All five CRD controllers are implemented and registered:

- **ClusterMember** — validates cluster identity (provider, region, clusterName) and sets `Ready=True/False` with an actionable status message
- **Federation** — resolves members via explicit list or label selector (or both), tracks per-member readiness, sets `Ready` and `Degraded` conditions
- **Workload** — reconciles a `Deployment` and `Service` in the local cluster; checks federation membership and `TargetPolicy` filters before acting; surfaces missing secrets as `Degraded/SecretNotFound` before creating any Deployment
- **ServiceEndpoint** — applies Cilium global service annotations (`service.cilium.io/global`, `service.cilium.io/shared`) based on `routingMode: ActiveActive` or `ActivePassive`; uses a finalizer to clean up annotations on delete
- **DatastoreBinding** — resolves a connection `Secret` and injects `DATASTORE_{NAME}_URL`, `DATASTORE_{NAME}_REPLICATION_MODE`, and (for bindings named `primary`) `DATABASE_URL` into the Workload's Deployment; uses a finalizer to remove injected env vars on delete

### Multicluster identity

The operator accepts a `--cluster-member-name` startup flag. When set, `WorkloadReconciler` checks that the local cluster is a member of the referenced `Federation` before reconciling runtime objects. Non-members receive a `Pending` target state and no Deployment is created. `TargetPolicy.members` and `TargetPolicy.memberSelector` provide additional placement filters.

### Secrets model

All secret references are local and namespace-scoped. The operator does not replicate secrets. Missing secrets produce actionable `Degraded` status with requeue. See `docs/architecture.md#secrets-model` for the recommended provisioning approach using External Secrets Operator.

### Local multicluster demo

A complete local demo creates two k0s clusters (`alpha`, `beta`), connects them with Cilium ClusterMesh, deploys the operator to each with the correct `--cluster-member-name`, and applies sample `ClusterMember`, `Federation`, `Workload`, and `ServiceEndpoint` manifests. See `examples/local-multicluster/README.md`.

### Repository validation

`bash scripts/validate-repo.sh` checks formatting, tests, required public docs, legacy/private reference absence, shell syntax, and optional CRD dry-run and kustomize rendering.

## What is explicitly out of scope for v0.1.0-alpha

- Database provisioning (DatastoreBinding assumes the database, role, and secret already exist)
- Secret replication across clusters (use External Secrets Operator or equivalent)
- Progressive rollout (canary, blue/green) — polykube is designed to compose with dedicated rollout controllers
- xCluster replication setup or management
- Gateway API integration for ServiceEndpoint (field accepted, not acted on)
- `ConflictPolicy` enforcement in DatastoreBinding (field accepted, informational only)
